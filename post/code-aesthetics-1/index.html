代码美学：为何要成为“不嵌套主义者”

2023-02-07

Contents

在 B站看到一个 up 主搬运的视频，题目是《代码美学：为何要成为“不嵌套主义者”》，观感很不错。就接着翻了一下这个作者的油管频道，将这个频道的内容记录到本文中，方便后续回顾。

Why You Shouldn’t Nest Your Code

“不嵌套主义者” 能接受的最大嵌套深度为 3，如果再多嵌套的话，可以想想 Linus Torvalds 的话。

if you need more than 3 levels of indentation, you’re screwed anyway, and should fix your program." - Linus Torvalds, Linux kernel coding style (partly justifying why the Linux kernel style uses 8-space indentation.

http://www.kernel.org/doc/Documentation/CodingStyle

那应该如何处理嵌套？

提炼（Extraction）：把复杂的逻辑提炼为一个单独的函数，这很容易理解，《重构》和《代码整洁之道》也有类似描述，每个函数应该只做一件事，保持每个函数内部处于一个抽象层级。

反转（Inversion）：the “checks and filters” block in the front of productive code. 把负面分支放在函数最前面，遇到了直接 return，后续再处理函数的正常流程。这类似于检查、过滤、拦截器等概念。比如下面这种函数形式。

func test () {
	// check and filters
	if xxx {
		return
	}
	
    // check 1
    if xxx {
		return
	}
	
	// productive code
}

降低嵌套层数的好处有很多，便于单元测试，提高代码可读性，可维护性。但要视情况而定，过度提炼会直接影响代码的可读性，为了消除嵌套，我们不得不提炼了很多使用次数很少的小函数，同时还伴随着命名不当的问题，这会极大的影响代码的可读性。

小结

降低嵌套的两种思路：提炼，保证每个函数内部尽可能位于同一抽象层次；反转，编写代码时，将错误路径和正常路径分离开。
